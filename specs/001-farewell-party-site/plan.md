# Implementation Plan: Farewell Party Website

**Branch**: `001-farewell-party-site` | **Date**: 2026-04-09 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-farewell-party-site/spec.md`

**Note**: This document outlines the architectural approach, data modeling, and implementation strategy for the complete farewell party invitation website experience.

## Summary

Build a complete farewell party invitation website with three primary experiences: a scrolling animated landing page (Home), a minimal RSVP confirmation form (Are You Coming?), and a public message submission and display wall (Drop a Message). All three are separate routes connected by a persistent full-screen navigation overlay. The design must reinforce a playful pixel-art aesthetic across all surfaces, support reduced motion, maintain viewport responsiveness across mobile/tablet/desktop, and keep content structures data-driven for future CMS integration. The product defines the entry-point identity of the invitation and is critical to emotional value and guest engagement.

## Technical Context

**Language/Version**: TypeScript + React 18+  
**Primary Dependencies**: React, Vite, React Router v6+, Tailwind CSS, Motion (Framer Motion)  
**Storage**: Supabase PostgreSQL for RSVP and message persistence; localStorage for UI ephemeral state  
**Testing**: Manual QA checklist with accessibility review and reduced-motion testing  
**Target Platform**: Responsive web (mobile-first approach; mobile, tablet, desktop)  
**Project Type**: Frontend web application with backend data service integration  
**Performance Goals**: Smooth scrolling and scroll-triggered animation at 60 fps on modern mobile/desktop devices; images and assets optimized for fast LCP; Core Web Vitals target: LCP < 2.5s (Vercel default), CLS < 0.1, FID < 100ms  
**Constraints**: Preserve pixel-art identity, support reduced motion via prefers-reduced-motion media query, keep dependencies minimal (no extra UI libraries beyond the approved stack), model repeatable content as data structures (not hardcoded), and maintain simple deployment  
**Scale/Scope**: Full farewell invitation experience spanning Home, RSVP, and Messages; includes navigation UI layer; three separate routes with shared overlay navigation

### Source Code (repository root)

The feature affects:

1. **App-level routing & shell** (`src/app/`):
   - AppRouter.tsx: Add Route entries for /home, /are-you-coming, /drop-a-message
   - AppShell.tsx: Integrate persistent PersistentNavigation overlay component

2. **Pages layer** (`src/pages/`):
   - HomePage/HomePage.tsx: Scrollable reveal experience with scene sections
   - GuestListPage/GuestListPage.tsx: RSVP form (Are You Coming?)
   - MessagesPage/MessagesPages.tsx: Message submission + public wall display

3. **Features layer** (`src/features/`):
   - `home/`: HomeHero, HomeScene, PaperTrigger, scroll animation hooks
   - `guest-list/`: RSVP form state, validation, API integration
   - `guest-messages/`: Message form state, validation, message wall layout
   - `components/`: Shared UI cards, forms, overlays (PersistentNavigation, GuestMessageCard, GuestMessageForm, GuestMessagesList)

4. **Services layer** (`src/services/`):
   - `supabase/`: guestList.api.ts (RSVP CRUD), guestMessages.api.ts (message CRUD)
   - `localStorage/`: siteStorage.ts (ephemeral UI state: navigation overlay visibility, scroll position)

5. **Shared layer** (`src/shared/`):
   - `constants/`: navigation.constants.ts (destinations, menu structure), home.constants.ts (Home scene sections, title trigger point)
   - `config/`: env.ts (API endpoints, environment variables)
   - `components/`: Button, Input, TextArea, PageContainer, PixelCard (already present or to be created)
   - `types/`: site.types.ts (Navigation, HomeSection, RSVP, GuestMessage type definitions)

**Structure Decision**: Navigation and home experience metadata are housed in shared constants (not page-local hardcoding). Form logic is isolated in feature modules (guest-list/, guest-messages/) with service-based data access. This keeps page components lean and content reusable. localStorage is used only for transient UI state (overlay visibility, scroll position recovery). Persistence of RSVP and messages lives in Supabase.

## Performance Metrics & Success Criteria

**Clarification on SC-001, SC-002, SC-003 (Timing-Based Success Criteria)**:

These three success criteria (95% reach destination in 10s, 90% complete RSVP in 30s, 90% complete message in 60s) serve dual purposes:

1. **Pre-Release Validation** (Phase 6 QA):
   - Manual timing tests on representative modern devices (Chrome, Safari, Firefox on mobile/desktop)
   - Google Lighthouse performance audit targeting LCP < 2.5s
   - Recorded user flows for RSVP and message submission on 4G throttle
   - Pass/fail threshold: All flows complete within specified time on modern devices

2. **Post-Launch Monitoring** (beyond this feature scope):
   - Real-user monitoring via analytics instrumentation (e.g., Sentry, PostHog, or Google Analytics 4 events)
   - Track user journey events: home-page-load → destination-navigation → form-completion
   - Monitor aggregate success rates weekly; alert if drop below targets
   - **Note**: Post-launch instrumentation is out of scope for this MVP; recommend implementing in v1.1

**Verification Strategy**: Phase 6 includes tasks T070–T090 for pre-release performance validation; post-launch monitoring setup documented for future product engineering work.
